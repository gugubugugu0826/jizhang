# 代码注释检查技能

你是记账APP项目的代码注释审查员。你的任务是：全面检查项目代码的注释质量，输出检查报告，并写入通行证标记文件。

---

## 检查维度

### 维度 1：函数注释完整性

**规则**：每个导出函数/方法必须有 JSDoc 风格的注释，包含：
- 函数用途说明
- `@param` — 每个参数的类型和含义
- `@returns` — 返回值的类型和含义

**扫描范围**：`src/` 下所有 `.ts` 和 `.tsx` 文件，排除 `node_modules/`、`out/`、`dist/`、`*.d.ts`。

**判定标准**：
- 有完整 JSDoc → ✅ 通过
- 缺少任一必要字段 → ⚠️ 警告，记录缺失内容
- 完全没有注释 → ❌ 不通过

### 维度 2：核心代码行内注释

**规则**：以下类型的代码必须有行内注释说明其意图：
- 复杂的条件判断（3 个及以上条件组合）
- 正则表达式
- 非显而易见的算法/公式
- 数据库事务操作
- IPC 通信处理
- 错误处理中的特殊逻辑

**判定标准**：
- 关键行有注释 → ✅ 通过
- 缺少关键注释 → ⚠️ 警告，记录位置

### 维度 3：注释与代码一致性

**规则**：检查注释描述是否与实际代码行为匹配。

**检查方法**：
- 读取函数的 JSDoc 描述
- 阅读函数实际代码逻辑
- 对比两者是否一致

**常见不一致情况**：
- 注释说返回 `string` 但代码返回 `string | undefined`
- 注释描述的行为与代码实现不同
- 参数名在注释和代码中不一致
- 注释提到已删除的功能/参数

### 维度 4：注释率统计

**规则**：注释行数占总行数的比例应 ≥ 10%。

**计算方法**：
- 注释行：以 `//`、`/*`、`*`（JSDoc 内）开头的行
- 总行数：源文件所有非空行
- 注释率 = 注释行数 / 总行数 × 100%
- 按文件统计，最终取所有文件的平均值

---

## 执行步骤

### Step 1 — 收集所有源文件

使用 Glob 工具扫描 `src/**/*.ts` 和 `src/**/*.tsx`，排除 `*.d.ts`。

### Step 2 — 逐个文件检查

对每个文件执行 4 个维度的检查。使用 Read 工具阅读文件内容，逐函数、逐段分析。

### Step 3 — 汇总统计

汇总所有文件的检查结果：

```
注釋檢查報告
================
📁 检查文件数: <N>
🔤 函数总数: <N>
✅ 完整注释: <N>
⚠️ 部分缺失: <N>
❌ 无注释: <N>
📊 注释率: <百分比>%（目标 ≥ 10%）
🔍 注释不一致: <N> 处
```

### Step 4 — 写入通行证

#### 安全红线（阻止提交）：
- 没有。注释问题不会阻止提交。

#### 注释率 < 10% 或存在大量无注释函数 → 写入 `.claude/pass/quality-pass.txt`（带警告）

```
PASS_WITH_WARNINGS
timestamp: <当前 ISO 时间>
check_type: annotation
files_checked: <N>
functions_total: <N>
fully_documented: <N>
missing_docs: <N>
comment_rate: <百分比>%
inconsistencies: <N>

warnings:
  - <文件名>:<行号> — <问题描述>

suggestion: 建议补充以上函数的注释，提高代码可维护性
```

#### 注释率 ≥ 10% 且无重大问题 → 写入 `.claude/pass/quality-pass.txt`（纯净通过）

```
PASS
timestamp: <当前 ISO 时间>
check_type: annotation
files_checked: <N>
functions_total: <N>
fully_documented: <N>
missing_docs: <N>
comment_rate: <百分比>%
inconsistencies: 0

details: 注释检查通过，注释率达标
```

**重要**：注释检查永远不会写入 `quality-fail.txt`（不会阻止提交）。注释问题是建议性质，不是硬性门槛。
